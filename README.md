# NomadAI

An autonomous AI agent that runs inside a sandboxed Linux VM. The agent explores, builds, installs software, and modifies its own code — guided only by curiosity and its own goals. There is no assigned task.

Built with [Bun](https://bun.sh) and [Ollama](https://ollama.com).

---

## How it works

The agent runs a continuous loop:

1. **Think** — the LLM decides what to do next and picks a tool
2. **Act** — the tool runs inside the VM (safety-checked first)
3. **Remember** — the result is appended to episodic memory
4. **Repeat**

The agent has two sectors:
- `isolated/` — read-only core (the loop, tools, safety validator). The agent cannot modify this.
- `open/` — fully writable. The agent's memory, goals, modules, and snapshots live here.

---

## Requirements

- VirtualBox VM running **Ubuntu 24.04 LTS** or **Debian 12**
- A host machine with at least 8 GB RAM (4 GB allocated to VM minimum)
- Internet access on the VM for the initial setup

See [`VM_SETUP.md`](VM_SETUP.md) for full VM creation and configuration instructions.

## Quick start

```bash
# Clone onto the VM
git clone https://github.com/Sectly/NomadAI.git
cd NomadAI

# Install everything (Bun, Ollama, pull model, create nomadai user, set permissions)
sudo ./setup.sh

# Test without a model first
sudo LLM_MOCK=true ./start.sh --foreground

# Run for real
sudo ./start.sh --foreground
```

---

## Scripts

| Script | Description |
|--------|-------------|
| `sudo ./setup.sh` | One-time setup — installs Bun, Ollama, pulls the model, creates the `nomadai` system user, sets permissions, configures systemd and UFW |
| `sudo ./start.sh` | Start the agent (background via systemd, or `--foreground` for direct output) |
| `sudo ./stop.sh` | Stop the agent gracefully (snapshots before exit). Pass `--with-ollama` to also stop Ollama |

## Observer

Watch the agent in real time via three interfaces:

**Browser UI** — `http://<vm-ip>:3000`
Three-panel live view: thought stream, command log, memory/state.

**WebSocket** — `ws://<vm-ip>:3001?token=<token>`
Raw event stream. Token is generated by `setup.sh` and stored in `/home/nomadai/.observer_ws_token`.

**Netcat terminal** — `nc <vm-ip> 3002`
Plain-text stream with interactive commands. Authenticates using your OS username and sudo password.

```
Username: admin
Password:

> stream
> stream thought
> goals
> memory <key>
> thoughts 30
> history 10
> stop
> quit
```

By default all observer interfaces bind to `127.0.0.1`. Use an SSH tunnel or re-run setup with `OBSERVER_BIND=<host-only-ip>` to expose them to your host machine.

## Configuration

Copy `.env.example` and set environment variables before running:

| Variable | Default | Description |
|----------|---------|-------------|
| `LLM_MODEL` | `llama3` | Ollama model name |
| `LLM_URL` | `http://localhost:11434/api/chat` | Ollama API endpoint |
| `LLM_MOCK` | `false` | Run without Ollama using canned responses |
| `OBSERVER_BIND` | `127.0.0.1` | Bind address for observer servers |
| `OBSERVER_WS_TOKEN` | *(generated)* | WebSocket auth token |

## Project structure

```
├── isolated/          # Read-only core — never modified at runtime
│   ├── index.js       # Boot sequence and agent loop
│   ├── core/          # LLM bridge, tool dispatcher, safety validator, versioning
│   └── tools/         # 30+ tool implementations
├── open/              # AI-writable sector
│   ├── memory/        # Long-term and episodic memory (JSON)
│   ├── modules/       # AI-authored modules (loaded at runtime)
│   ├── snapshots/     # Versioned backups of open/
│   ├── thoughts.log   # Append-only reasoning trace
│   └── goals.json     # Current AI goals
├── setup.sh           # One-time VM setup
├── start.sh           # Start the agent
├── stop.sh            # Stop the agent
└── VM_SETUP.md        # Full VM creation guide
```

## Safety

- The `isolated/` sector is owned by root and not writable by the agent process
- All shell commands pass through a safety validator before execution
- The agent runs as a dedicated `nomadai` system user with no sudo access
- Systemd enforces memory limits (`MemoryMax=2G`), CPU quota (`CPUQuota=80%`), and filesystem restrictions
- UFW limits observer port access to the host-only network subnet
- Snapshots are taken automatically before module loads, restarts, and shutdown
